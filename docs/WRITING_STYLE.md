# Стиль текста

Этот репозиторий читают разработчики. Пишите так, будто объясняете коллеге, что
есть в проекте и где границы. Без витрины. Без внутренней переписки с владельцем.

## Базовые правила

- Одна мысль на предложение.
- Уберите слово, если без него смысл не меняется.
- Не обещайте больше, чем репозиторий делает.
- Не пишите "лучший", "современный", "production-ready", если дальше нет
  проверки или ограничения.
- Английский термин оставляйте, если он так используется в коде или документации:
  `hook`, `fixture`, `failure card`, `checker`.
- Если есть сомнение, так и пишите. Лучше честная граница, чем уверенный шум.

## Запрещенные паттерны

Не используйте такие фразы:

- "помогает писать хороший код";
- "делает backend безопасным";
- "универсальное решение";
- "просто подключите и получите production quality";
- "стоит отметить";
- "в заключение";
- "погрузимся";
- "раскрывает потенциал".

Обычно они скрывают отсутствие конкретики.

## README

README должен быстро отвечать на вопросы:

1. Что это?
2. Когда это брать?
3. Что внутри?
4. Чем это проверено?
5. Как установить?
6. Где границы?

Не кладите в README внутренние TODO, handoff notes и личные напоминания.
Для этого есть issues, backlog или `docs/CLAUDE_HANDOFF.md`.

## Документация

Каждый абзац должен делать одну из трех вещей:

- объяснять решение;
- задавать правило;
- фиксировать доказательство.

Если абзац только звучит умно, удалите его.

## Комментарии в коде

Комментируйте причину, границу или неочевидный tradeoff.

Плохо:

```python
# Read the file.
text = path.read_text()
```

Лучше:

```python
# Forward-test prompts must stay self-contained; broken prompts hide skill regressions.
text = path.read_text()
```

Комментарии `PLANTED:` и `CLEAN:` в fixtures допустимы. Они часть тестового
контракта.

## Запрос для агента

Перед правкой публичного текста можно дать агенту такой короткий prompt:

```text
Перепиши текст как живой README или developer doc.
Пиши коротко. Одна мысль на предложение.
Убери маркетинг, внутренние TODO и фразы без проверяемого смысла.
Не обещай больше, чем проект делает.
Сохрани техническую точность, ссылки и команды.
Если текст звучит как шаблон, перепиши еще раз.
```
