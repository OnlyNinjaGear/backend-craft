# Анатомия трёх дизайн-скиллов — фундамент для бэкенд-пакета

Дата разбора: 2026-07-09. Исходники лежат в `_reference/` (impeccable, taste-skill, emilkowalski-skills).
Каждый репозиторий прочитан агентом целиком; ниже — синтез.

---

## 1. Общая формула: почему эти скиллы работают

Все три пакета, независимо друг от друга, сходятся в одном диагнозе и одном лечении.

**Диагноз:** модель плоха не потому, что не знает, а потому что тянется к своим
статистически самым вероятным дефолтам (slop). Она срезает шаги, рапортует об успехе
без проверки, соглашается с собственным выводом.

**Лечение — пять столпов:**

1. **Назвать дефолты модели по имени и запретить их списком.** Не «избегай плохого кода»,
   а конкретная сигнатура + почему это tell + чем заменить. Формат «Match-and-refuse»:
   «если собираешься написать вот это — перепиши вот так».
2. **Числа вместо прилагательных.** Не «делай функции небольшими», а таблицы бюджетов
   с конкретными диапазонами. Не «спроси при неясности», а «ровно один вопрос, и только если
   прочтения реально расходятся».
3. **Принудительный порядок работы.** Обязательный setup («это не опционально; пропуск даёт
   generic-результат»), гейты, которые «нельзя сжимать», обязательное однострочное объявление
   о каждом пропущенном шаге.
4. **Финальный механический чек-лист.** «Если хоть одна галочка не ставится честно — работа
   не закончена». Ответ на каждый пункт — цитата файла/строки/значения, никогда голое «да».
5. **Инженерия против собственных сбоев модели.** Правила прямо называют, как модель
   исторически нарушала их: «агент игнорировал лимит на em-dash при формулировке "use
   sparingly" — поэтому формулировка бинарная: ноль em-dash». Скилл — это регрессионный
   набор против известных отказов модели.

## 2. Три пакета — три архитектуры

### impeccable (Paul Bakaus) — «комбайн»
- **Hub-and-spoke:** один SKILL.md (~22KB: законы + таблица 23 команд + роутинг),
  32 reference-файла грузятся строго по требованию. Общий корпус ~400KB, в контексте — только нужное.
- **Скрипты как второй слой:** 46 анти-паттернов как *данные* (не проза), 4 движка детектора,
  PostToolUse-хук, который после каждой правки шепчет модели bounded-напоминание
  (max 5 находок, дедуп, «never break a turn, always exit 0»).
- **Регистры:** каждая задача — «brand» или «product», у каждой команды противоположные
  советы в зависимости от регистра. Одна команда — два свода правил.
- **Двойная изолированная оценка:** LLM-ревью и детектор бегут параллельно в изоляции,
  чтобы вывод сканера не «заякорил» суждение. Деградация — только с баннером
  «⚠️ DEGRADED» первой строкой.
- **Проектная память:** PRODUCT.md (стратегия) + DESIGN.md (токены) — всё читается перед работой.
- **Скилл сам покрыт тестами:** LLM-сценарии проверяют по трейсу тул-коллов, что setup
  реально выполняется. «Трейс — источник истины, а не свободный ответ модели».

### taste-skill (leonxlnx) — «жёсткая книга правил»
- 13 однофайловых скиллов, режимы = отдельные скиллы (default / строгий для GPT /
  redesign / анти-обрезание вывода), а не подкоманды.
- **Дозируемость через числовые «диалы»** (1–10) с таблицей вывода диала из брифа.
- **Каталог банов, выведенный из реальных провалов:** каждый бан цитирует точный артефакт
  из продакшн-тестов. CHANGELOG документирует ужесточение каждого правила.
- **Правило-шаблон:** «X запрещён как дефолт. Override: X допустим, когда <условие>».
  Бинарные запреты без override — только там, где мягкая формулировка доказанно не работала.
- **Пре-флайт из ~60 галочек** с формулами («count > ceil(sectionCount/3) → fail»).
- **Канонические скелеты кода** для эмпирически ломающихся паттернов, с пометкой
  «Common failure → Fix».
- Ноль скриптов — но правила написаны так, что детектор по ним пишется тривиально.

### emilkowalski-skills — «вкус как личность»
- 4 маленьких скилла, у каждого одна работа. Знания / ревьюер / словарь / чужой канон.
- **Split «процесс/планка»:** SKILL.md ревьюера — метод (112 строк), STANDARDS.md — точные
  значения («цитируй их, не аппроксимируй»). Убивает галлюцинацию правдоподобных чисел.
- **Поза ревьюера:** «Default to flagging. Approval is earned, not assumed» +
  `disable-model-invocation: true` — агрессивный ревьюер безопасен только потому,
  что не может самозапуститься посреди работы.
- **Лестница исправлений delete-first:** «1. Удали анимацию. 2. Уменьши. … 8. Полируй».
  Минимализм как механический tiebreaker.
- **Таблицы дозировки:** «элемент виден 100+ раз в день → никакой анимации. Ever.»
- **Каждое правило = тройка:** императив + механизм-почему (одно предложение) + Bad/Good-пара.
- **Военные истории как авторитет:** «на Vercel таб-анимация роняла кадры — перешли на CSS».

## 3. Каталог приёмов (что заимствуем дословно)

### Структура
- Hub-and-spoke + progressive disclosure; общие законы всегда в контексте, глубина — по требованию.
- Split «метод / планка значений» (SKILL.md / STANDARDS.md).
- Знания-монолит отдельно от ревьюера; ревьюер — только ручной запуск.
- Финальная Quick Reference-таблица (Need | Technique | Value), сжимающая весь файл в ~20 строк.

### Формулировки правил
- Атомарная единица: **императив + механизм + Bad/Good-пара + escape hatch**.
- Именованные правила («THE LILA RULE», «The One Voice Rule») — цитируемые ручки.
- Запреты с точными сигнатурами кода, не с прилагательными.
- Исключения повторяются **у каждого** вхождения родительского правила.
- «Correct-the-textbook»: предвосхитить самый вероятный правдоподобно-неверный ответ модели.
- Бинарные формулировки там, где мягкие доказанно игнорируются, — с указанием этой истории в самом правиле.

### Процесс
- Обязательный setup-контракт с «почему пропуск фейлит» на каждом шаге + идемпотентность.
- Forced «Design Read» → у нас «Task Read»: одно предложение интерпретации до кода.
- Гейты с анти-компрессией: «одобрение шага N не открывает шаг N+2».
- Обязательное объявление пропуска: «шаг X пропущен, потому что Y» — одной строкой, всегда.
- Вопросы: 2–3 за раунд, assert-then-confirm, «если рядом с вопросом написал бы Recommend: X — просто реши X».
- Cost-asymmetry-развилки: выписать цену обеих ошибок, дефолт — восстановимая («не уверен — ты в default mode»).

### Проверка
- Cite-or-fail: каждый пункт чек-листа закрывается цитатой (файл:строка, имя теста, число бенчмарка).
- «Чистый скан — пол, а не вердикт». Вывод детектора — улики, не приговор.
- Анти-фейк: «не выдумывай дефекты ради видимости итерации», «прогон тестов, чей вывод ты не прочитал, не считается».
- «Отсутствие тула — не разрешение пропустить: найди рантайм или остановись и доложи».
- Скоринг: рубрики 0–4 по измерениям, severity P0–P3 с tiebreaker'ом, снапшоты + тренд между прогонами.

### Активация
- Description: плотный список глаголов-триггеров + перефразированные интенты + **негативный скоуп** («Not for…»).
- 2–3 самых нарушаемых правила — прямо в description (переживают description-only загрузку).
- `disable-model-invocation: true` для агрессивных ревьюеров.
- Worked examples вход→выход прямо в description (учат роутер демонстрацией).

### Голос
- Императив, второе лицо, короткие приговоры («Cards are the lazy answer.»).
- Сила модального глагола = сила правила (MUST/Never vs prefer/consider) — последовательно.
- О модели — в третьем лице («LLMs are inherently lazy») — compliance как преодоление собственного bias.
- Зоны вкуса помечены явно («формулы нет — подбирай»), отчего всё непомеченное читается как обязательное.

## 4. Чертёж бэкенд-пакета (предложение)

Рабочее имя пакета: **solid-backend** (обсуждаемо). Состав — зеркало дизайн-тройки:

```
solid-backend/
├── skills/
│   ├── backend-philosophy/SKILL.md      # аналог emil-design-eng: всегда-на подходе
│   │                                     # инварианты → границы ошибок → данные → код
│   ├── python-backend/SKILL.md          # идиомы, скелеты, баны + STANDARDS.md
│   ├── go-backend/SKILL.md              # то же для Go
│   ├── ts-backend/SKILL.md              # то же для Node/TS
│   ├── review-backend/                  # аналог review-animations:
│   │   ├── SKILL.md                     #   метод, disable-model-invocation: true
│   │   └── STANDARDS.md                 #   планка: таймауты, ретраи, пулы, точные значения
│   └── backend-vocabulary/SKILL.md      # словарь: «когда два деплоя бегут вместе» → blue-green/canary
├── detector/                            # фаза 2: баны как данные + скрипт + PostToolUse-хук
└── README.md
```

**Ядро содержимого (выведено из разбора):**
- «AI Tells» бэкенда: `except Exception: pass`, goroutine без context/errgroup, HTTP-клиент
  без таймаута, `any` в публичном API, SQL-конкатенация, N+1 в цикле, ретрай без джиттера,
  `time.sleep` в async, `utils.py`-свалка, фейковые метрики в докстрингах, `// TODO: handle error`.
- Дозировка по критичности пути: деньги/авторизация → полная матрица кейсов (успех, ошибка,
  граница, права, восстановление); hot path → без аллокаций/рефлексии; одноразовый скрипт → можно проще.
- Лестница исправлений: 1. Удали код/абстракцию → 2. Возьми stdlib/существующий хелпер →
  3. Чини владеющий слой → 4. Упрости поток данных → 5. Добавь таймаут/контекст → 6. Только потом оптимизируй.
- Гейт перед кодом: «А нужна ли вообще эта абстракция/сервис/горутина/кэш?» — вопрос №1 может убить работу.
- Dependency-гейт: «перед импортом любой библиотеки проверь манифест; нет — сначала команда установки».
- «What Never Changes Silently»: пути эндпоинтов, схемы ответов, семантика колонок БД,
  имена метрик/событий, формат логов, на который кто-то подписан.
- Финальный пре-флайт: типы, тесты изменённой поверхности, error paths, обратимость миграций,
  cite-or-fail на каждый пункт.

## 5. Дальнейшие шаги

1. **Фаза 1 — философия + ревьюер** (самый быстрый эффект): backend-philosophy и
   review-backend + STANDARDS.md. Ревьюер — строго ручной запуск.
2. **Фаза 2 — три языковых скилла**: по каталогу банов + канонические скелеты на каждый
   эмпирически ломающийся паттерн (context/errgroup, async-отмена, идемпотентные миграции,
   транзакции, валидация на границе).
3. **Фаза 3 — детектор + хук**: перевести баны из прозы в данные, скрипт-сканер,
   PostToolUse-хук по образцу impeccable (bounded, дедуп, always exit 0).
4. **Фаза 4 — испытание на живом проекте** (бот «Джанго» — идеальный полигон)
   и пополнение каталога tell'ов по каждому реальному провалу, CHANGELOG-style.
