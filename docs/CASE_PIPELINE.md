# Как пополнять скилл реальными кейсами

Реальные проекты полезны не потому, что в них все правильно. Наоборот. Они дают
сырье: баги, ревью-комментарии, странные edge cases, инциденты и промахи агентов.

В скилл попадает не весь кейс. Попадает только выжатый из него повторяемый
failure mode.

## Вход

Подходит:

- баг из реального backend-проекта;
- PR, где ревью поймало опасную правку;
- инцидент или postmortem;
- ошибка агента на задаче;
- официальный документ, который меняет конкретное правило.

Не подходит:

- "давайте добавим раздел про Kafka";
- "надо больше best practices";
- "вот большой проект, пусть Claude найдет что-нибудь";
- статья без конкретного failure mode.

## Выход

После разбора должен появиться один из артефактов:

| Артефакт | Когда нужен |
|---|---|
| Failure card | ошибка повторяется и ее можно объяснить коротко |
| Verifier | проблему можно доказать тестом, diff, запросом, логом или contract check |
| Checker | есть надежный механический паттерн для Semgrep/ast-grep/hook |
| Playbook step | правило важно, но ловится только процедурой |
| Rejection note | кейс интересный, но пока не дает проверяемого материала |

Если нет ни одного артефакта, кейс не попадает в скилл.

## Процесс для Claude

1. **Анонимизировать.** Убрать названия компаний, домены, токены, PII и лишний
   код.
2. **Сжать.** Оставить минимальный пример, где ошибка еще видна.
3. **Сопоставить.** Найти существующую failure card или объяснить, почему нужна
   новая.
4. **Доказать.** Предложить verifier. Без verifier материал остается в backlog.
5. **Проверить источником.** Сослаться на официальный документ, если правило не
   пришло из наблюдаемого поведения кода.
6. **Сделать слепой тест.** Дать свежему агенту задачу без подсказки ожидаемого
   ответа.
7. **Принять решение.** `reject`, `backlog`, `fixture-tested` или
   `production-tested`.

## Шаблон запроса

```text
Работай как evidence distiller для backend-craft.

Разбери приложенный backend-кейс.
Не добавляй советы прямо в skill.

Сделай:
1. короткое описание проблемы без приватных данных;
2. минимальный reducer или объясни, почему reducer невозможен;
3. mapping к существующей failure card или proposal новой card;
4. verifier: test, checker, query, log assertion или review step;
5. source backing, если нужен внешний источник;
6. blind forward-test prompt;
7. promotion decision: reject / backlog / fixture-tested / production-tested.

Запрещено добавлять общие best practices.
Материал принимается только если дает card, verifier, checker или source-backed step.
```

## Планка promotion

| Статус | Что значит |
|---|---|
| `reject` | кейс не дает повторяемого правила |
| `backlog` | есть сигнал, но нет reducer или verifier |
| `fixture-tested` | правило поймано на искусственном примере |
| `production-tested` | правило подтвердилось на реальном backend-коде или слепом тесте с похожим кейсом |

Не повышайте статус за красивое объяснение. Повышайте только за проверку.
